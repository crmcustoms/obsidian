## Що зроблено

Python-сервіс що замінив нестабільні n8n-воркфлоу для виgрузки фінансових даних з Finansist в Google Sheets.

**Результат першого запуску:**
- 📋 ПЛАН (заявки на платіж): **~480** операцій
- ✅ ФАКТ (фактичні платежі): **~26 500** операцій
- Охоплення: всі роки в системі

**Технічний стек:**
- Python 3.12 + Docker
- httpx + tenacity (retry)
- gspread (Google Sheets API)
- SQLite (стан синхронізації)
- Cron: 3 рази на день (7:00, 14:00, 19:00 UTC)

**Сервер:** Linux Fedora (`fedvypuxfn`)
**Репо:** https://github.com/crmcustoms/apexgooglesheets
**Google Sheets:** https://docs.google.com/spreadsheets/d/1Ar9XQMk-AEiv1ZZDiZXBPvGiTV8lkt0behN6xh53DgY/

## Ключові рішення

- Вкладки по роках (2022, 2024, 2025, 2026, 2027...) — створюються автоматично
- 30 колонок на операцію (ID, тип, дата, сума числом, статья, проєкт, контрагент, відповідальний...)
- Інкрементальна синхронізація: дублі не створюються, зміни будь-якого поля → рядок оновлюється
- При `--force`: листи видаляються і створюються заново (вирішення ліміту 10M комірок Google)

## Проблеми під час розробки та вирішення

| Проблема | Рішення |
|----------|---------|
| Finansist API: statusId обов'язковий для PaymentRequests | Передаємо всі статуси [710,711,712,713] |
| Дублі після кожного запуску | Баг: порівнювали `"plan"` з `"план"` — виправлено |
| Ліміт Google Sheets 10M комірок | delete+recreate листа замість clear+resize |
| Суми як текст | Передаємо float/int напряму |
| Відповідальний — порожній | Поле `userName` напряму з запису (не через lookup) |
